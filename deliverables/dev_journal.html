Building smart-ALEKS
Sherman × Claude · A three-day collaborative build log

This document narrates the full development arc of smart-ALEKS: from an empty repo to a working extraction pipeline, and then — after discovering that pipeline no longer matched reality — to a cleaner architecture that reflects how the work is actually done. Sherman provided direction, domain context, and quality judgment; Claude handled code generation, schema design, and documentation.

Day 1 · June 23, 2026 · ~4:45 – 6:00 PM
First Claude Session — Schema, Skill, and the QA Gate
~1.25 hr real time
~25 min compute
~18,000 output tokens
Sherman + Claude

The first real Claude session. Sherman brings the extraction problem in full: ALEKS PDFs are dense chart-heavy dashboards. He knows the target variables — effect sizes, subgroup breakdowns, usage compliance, ESSA evidence tiers — and he wants a Pydantic schema to validate whatever comes out of the LLM, and an HTML QA gate so Amanda can review every value before it reaches district leadership.

In a single session Claude produces rce_schema.py — three nested Pydantic v2 models: RCEReport, UsageAnalysis, and SubgroupFinding — with CORE/EXT/META field tiers, a custom validator that parses raw effect-size strings, and enum types for ESSA tier, effect direction, and statistical significance. The CORE/EXT/META tier pattern is established here and holds for the rest of the project.

SKILL.md is also written: the extraction protocol, giving the LLM sequential chart-reading rules to combat spatial drift on bar charts. field_source_map.md maps every schema field to its visual location in the PDF. And a first draft of spot_check.html arrives — a self-contained browser QA gate with confidence tiering, ambiguity flags, and export. All of it lands in a staging folder: contributors/sherman/_RCE_Contrib/.

Design decision locked here: the QA gate must be a self-contained HTML file with no server dependency. Amanda can open it locally, drag-drop a JSON, and review without installing anything. This constraint shapes every subsequent output artifact.

Day 2 · June 24, 2026 · ~1:30 – 5:45 PM
The PDF Experiment, Spatial Drift, and the Pivot to Gemini XLSX
~4.25 hr real time
~50 min compute
~35,000 output tokens
Sherman + Claude

The afternoon of Day 2 is where the project learns something important.

Sherman uploads the five de-identified ALEKS PDFs. Claude builds extract_rce.py: a script that reads each PDF page as an image and sends it to Claude Vision with the SKILL.md extraction prompt, expecting a structured RCEReport JSON back. It has retry logic, schema validation, and a working prompt. On paper it is the cleanest possible pipeline: PDF in, validated JSON out.

The problem: ALEKS PDFs are image-only exports — each page is an embedded JPEG with no text layer. That is manageable. What is not manageable is spatial drift: the callout labels on ALEKS bar charts are printed in a column that is visually offset from the bars they label. Claude Vision reads them left-to-right and maps each label to the nearest bar — which is consistently the wrong bar. Gender subgroup values swap. Ethnicity rows shift by one. The extractions look plausible but are systematically wrong in a way that is hard to catch without the source PDF open beside them.

The fix is not a better prompt. It is a different tool. Amanda, the senior researcher, had already been using Gemini Enterprise for extraction in parallel and producing a verified XLSX. The architecture needed to reflect that reality: Gemini extracts, P2 manually QAs, the XLSX is the trusted output.

Pivot decision: abandon Claude Vision as the primary extractor. extract_rce.py stays in the repo as a record of what was tried but is no longer the path. All extraction routes through Gemini Enterprise → XLSX → human review.

xlsx_to_rce.py is built next: a converter that reads Amanda's Gemini XLSX, maps 17 subgroup finding rows, the overall effect size, usage compliance, and fidelity threshold, and emits a Pydantic-valid RCEReport JSON. spot_check.html gets its drag-drop JSON loader. populate_spot_check.py is added so the CLI can pre-populate a review file without ever touching the QA prototype directly (a lesson in accidental overwrites). artifact.html is scaffolded as the final output target — an interactive leader brief for district decision-makers. The README is updated.

By 5:45 PM the MVP pipeline exists in code: XLSX in, JSON out, QA gate, HTML artifact. Sherman pushes. On paper this is the finish line for Phase 3.

Day 3 · June 25, 2026 · ~4:35 – 5:40 PM
Pipeline Redesign, Cleanup, and Closing the Gap
~1.75 hr real time
~35 min compute
~24,000 output tokens
Sherman + Claude

This is the session that rewrites the pipeline.

Sherman returns with a clearer picture of how the work is actually flowing. Amanda and Agent 2 are not just running synthesis — they are building a Client-Specific Agent per engagement, one configured with the current XLSX and prior research briefs to do longitudinal contextual analysis. The output is a Google Doc, not an HTML file. xlsx_to_rce.py and the RCEReport JSON intermediary never made it into the live workflow. The Research Brief Agent instruction template — a full operating spec for the synthesis agent, with anomaly detection patterns, output structure, evidence rules, and a self-verification checklist — had been written by Sherman and was sitting in docs/ waiting to be connected to the rest.

The flowchart is rebuilt to match reality: an Agent 1 correction loop (P2 routes errors back to Gemini for a corrected XLSX, which re-enters review), a new Client-Specific Agent node, Previous Research Briefs feeding in from the left for longitudinal context, and the final output relabeled as Google Doc. The tooltip descriptions are rewritten throughout.

The two-year problem and the thirty-minute fix: the gap between what the codebase implied and what the pipeline actually was had been growing for two days of parallel development. The code said "JSON → spot_check → artifact.html." The actual workflow said "XLSX → Gemini → Google Doc." The flowchart made the gap visible for the first time.

The cleanup follows. Claude audits the full file tree — 28 tracked files, two experimental branches, five PDFs committed against the gitignore rules — and presents a phased plan. Sherman confirms which contributor directories to leave alone and greenlights the approach. A cleanup/reorganize branch is created; four commits run in under two minutes of wall-clock time:

Phase 1: untrack the PDFs and the JSON gold fixture that had been committed before the gitignore rules were added. Phase 2: remove spot_check.html, artifact.html, rce_schema.py, scaffold_project.sh, VALIDATION_SKILL.md, status_report.py, and the empty test placeholder. Phase 3: remove xlsx_to_rce.py, extract_rce.py, populate_spot_check.py, and the entire _RCE_Contrib staging folder from contributors/sherman/. Phase 4: rewrite CLAUDE.md and README.md to describe the pipeline that actually exists.

The branch merges to main. llm-direct and meta-analysis-full — the experimental branches that had never been pushed to remote — are deleted. The repo ends where the pipeline actually is.
